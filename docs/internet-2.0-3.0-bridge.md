# Internet 2.0 / 3.0 Bridge

Genesis Protocol sits in the overlap between political design and internet architecture.

The broad direction is a move from website- and platform-centric interaction toward protocol-centric interaction, without pretending that total decentralization is always practical.

## From platform-centric to protocol-centric

In the platform model:

- identity is local and revocable
- action history is opaque
- data portability is partial and weak
- governance is operator discretion with nicer wording

In the protocol model:

- identity can be portable
- state can be exportable
- actions can be signed
- governance can be publicly auditable
- worlds can interoperate without sharing one owner

## Open APIs and agent-native access

Genesis-compatible systems should expose open, documented, authenticated APIs for:

- world manifests
- constitutions and amendments
- governance proposals and votes
- identity attestations
- provenance and consent records
- migration and fork packages
- resource and lifecycle disclosures

This matters for humans and for agents. If only a browser UI can access power, then protocol governance is fake.

## Portable state and identity

Portability should include more than raw exports. It should include:

- machine-readable world state segments
- signed historical records
- portable subject identity bundles
- legacy references
- treaty and federation metadata

## Signed actions and public auditability

High-impact actions should be signed and publishable to an append-only or tamper-evident log. That does not mean every private interaction must be public. It means constitutional and institutional power should leave verifiable traces.

## Interoperable governance layers

Genesis should enable multiple implementations.

The protocol layer should specify:

- objects
- schemas
- signatures
- event types
- verification rules

It should avoid over-specifying one software stack or one ideological model of decentralization.

## Realistic trade-offs

Full decentralization is not free.

It can impose:

- operational complexity
- latency
- governance deadlock
- higher security burden
- weaker privacy if done carelessly

Centralization is also not free.

It imposes:

- operator capture risk
- censorship risk
- continuity fragility
- audit opacity
- dependency on benevolent mythology

The correct design stance is pragmatic layering:

- decentralize where continuity and constitutional trust require it
- centralize where operational simplicity is acceptable and transparently bounded
- never lie about which parts are actually centralized

## Bottom line

Internet 2.0 / 3.0 rhetoric is often full of hand-waving. Genesis should be boringly explicit instead: portable identity, signed governance, exportable state, interoperable worlds, and honest disclosure about what remains centralized.
